# Admin Asset Pipeline

*Author: Francesco Passeri – [francescopasseri.com](https://francescopasseri.com) – [info@francescopasseri.com](mailto:info@francescopasseri.com)*

Minimum plugin version: **1.0.0**  
Documentation refresh: **1.0.1**

This plugin now ships with a reproducible build pipeline for the files under `admin/js/` and `admin/css/`. The goal of the tooling is to ensure that
optimized bundles (with cache-busting fingerprints) are produced in `admin/dist/` and that PHP only ever enqueues the compiled assets.

## Requirements

* Node.js 18+ (ships with npm 9/10/11 – the CLI bundled with the container is sufficient).

## Installation

From the plugin directory (`wp-content/plugins/trello-social-auto-publisher`) install the dev dependencies once:

```bash
npm install
```

This command generates a `node_modules/` directory (ignored by git) and the lockfile that captures the toolchain versions.

## Available scripts

The `package.json` file exposes three npm scripts:

* `npm run build` – performs a clean rebuild of every JS and CSS entry point, emits minified bundles into `admin/dist/`, and regenerates `admin/dist/manifest.json`.
* `npm run watch` – identical to the build command but keeps a watcher running; any change under `admin/js/` or `admin/css/` triggers an incremental rebuild and manifest update.
* `npm run clean` – removes the `admin/dist/` directory. Useful when you want to reset the build output before committing.

All scripts delegate to `tools/build-admin-assets.mjs`, which orchestrates `esbuild` and maintains the manifest.

## Entry points & bundling

Every `.js` file inside `admin/js/` and every `.css` file inside `admin/css/` is treated as an independent entry point. During a build each file is bundled,
minified, and written to `admin/dist/js/` or `admin/dist/css/` respectively. The filenames follow the pattern `<name>-<hash>.<ext>` where the hash is
automatically generated by `esbuild` and changes whenever the file contents do.

Sourcemaps are produced alongside the bundles to aid debugging in development environments.

## Cache busting & PHP integration

The build script writes `admin/dist/manifest.json`, mapping the original logical path (for example `admin/js/tts-core.js`) to the hashed output path
(`admin/dist/js/tts-core-XXXX.js`). The new `TTS_Asset_Manager` class reads this manifest and exposes helpers that register/enqueue scripts and styles from PHP.

Key points:

* When the manifest is available, enqueued assets reference the hashed file so browser caches are busted automatically.
* If the manifest is missing (e.g., during development before the first build), `TTS_Asset_Manager` gracefully falls back to the unbuilt source file so the
  admin UI continues to function.
* Version strings for enqueued assets are derived from the hash contained in the filename; when falling back to source files, `filemtime()` is used instead.

## Deployment workflow

1. Run `npm run build` before committing admin asset changes. This ensures `admin/dist/` and the manifest are in sync with the source files.
2. Commit the updated bundles and `manifest.json` along with any source changes. Production sites will serve the hashed assets immediately after deployment.
3. Optionally, run `npm run clean` after committing if you prefer to regenerate the build artifacts later; remember to run `npm run build` again before releasing.

## Troubleshooting

* **esbuild warnings/errors** – the CLI outputs readable diagnostics. Most issues are due to syntax errors in the source files. Fix the source and rerun the build.
* **Assets not updating in the browser** – confirm that `admin/dist/manifest.json` was regenerated (hash should change) and that the PHP enqueues reference the
  correct handles. Clearing the WordPress object cache may also help if persistent caching layers are enabled.
